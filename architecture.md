# Pomodoro Timer Web App Architecture

## 1. 目的

このドキュメントは、Flask と HTML/CSS/JavaScript を用いてポモドーロタイマーの Web アプリを実装するためのアーキテクチャ案を整理したものです。

UI モックは単一画面で完結する構成であり、集中タイマー、円形プログレス、開始・リセット操作、今日の進捗表示が中心です。そのため、初期実装では Flask を薄く保ち、タイマーの状態管理と画面更新はフロントエンド側で完結させる構成を採用します。

## 2. 設計方針

- Flask は画面配信と将来の API 拡張のための受け口に限定する
- タイマーの進行ロジックはブラウザ側で管理する
- UI とロジックを分離し、ユニットテストしやすい構造にする
- 初期版では localStorage を利用して設定値と日次進捗を保存する
- 将来的に SQLite を追加しやすいよう、保存処理は抽象化する

## 3. 全体アーキテクチャ

アプリケーションは以下の 4 層で構成する。

### 3.1 プレゼンテーション層

責務:

- Flask のルート定義
- HTML テンプレートの返却
- CSS による見た目の構築
- JavaScript からの DOM 更新

役割:

- 画面の表示
- ユーザー操作の受付
- アプリケーション層へのイベント委譲

### 3.2 アプリケーション層

責務:

- 開始
- 一時停止
- リセット
- セッション完了時の次モード決定
- 進捗更新の調停

役割:

- プレゼンテーション層からのイベントを受ける
- ドメイン層のロジックを呼び出す
- 保存処理の呼び出しを調停する

### 3.3 ドメイン層

責務:

- タイマー状態の定義
- モード遷移ルールの定義
- 残り時間計算
- 集中完了回数と累積時間の集計

役割:

- ビジネスルールを持つ中心層
- 副作用を持たない純粋ロジックを優先する

### 3.4 インフラ層

責務:

- localStorage への保存と読込
- 将来の SQLite 保存
- 通知音やブラウザ通知などの外部 API 呼び出し

役割:

- 永続化と外部依存を吸収する

## 4. Flask とフロントエンドの責務分離

### 4.1 Flask 側で行うこと

- 初期画面の返却
- 静的ファイル配信
- 必要であれば初期設定値の受け渡し
- 将来の JSON API 提供

### 4.2 フロントエンド側で行うこと

- タイマー状態の保持
- 1 秒ごとの再描画
- 円形プログレスの更新
- ボタン操作による状態変更
- localStorage への保存
- セッション完了通知

このアプリでは、サーバー側が毎秒タイマー状態を持つ必要はありません。単一ユーザーのブラウザ内で完結するため、クライアント主導の構成が最もシンプルです。

## 5. タイマー状態設計

タイマーは単純に残り秒数を毎秒減算するのではなく、終了予定時刻を基準に残り時間を再計算する方式を採用します。

理由:

- タブ切り替え時のずれを減らせる
- 一時停止と再開の扱いが明確になる
- テスト時に時刻を固定しやすい

### 5.1 主要な状態

- 現在モード
  - work
  - short_break
  - long_break
- 稼働状態
  - idle
  - running
  - paused
- 開始時刻
- 終了予定時刻
- 残り秒数
- 完了した集中回数
- 今日の累積集中時間

### 5.2 設定値

- 集中時間
- 短休憩時間
- 長休憩時間
- 何回ごとに長休憩へ移行するか

## 6. 推奨する主要コンポーネント

### 6.1 TimerState

責務:

- 現在のモードと時間情報を保持する
- UI 表示に必要な値の基礎データを持つ

### 6.2 TimerEngine

責務:

- 開始
- 一時停止
- 再開
- リセット
- 残り時間の再計算
- セッション完了判定

設計方針:

- 可能な限り純粋関数か小さなクラスとして実装する
- DOM や localStorage には直接依存しない

### 6.3 SessionPolicy

責務:

- work 完了後に short_break へ進むか long_break へ進むかを決定する

### 6.4 ProgressService

責務:

- 集中完了回数の更新
- 今日の累積集中時間の更新
- 日付変更時のリセット判定

### 6.5 SettingsRepository

責務:

- 設定値の保存と読込

### 6.6 ProgressRepository

責務:

- 今日の進捗データの保存と読込

## 7. UI 実装方針

画面は単一ページで構成する。

### 7.1 画面要素

- ヘッダー
  - アプリ名
  - ウィンドウ風の装飾
- タイマー表示領域
  - 現在モード
  - 残り時間
  - 円形プログレス
- 操作領域
  - 開始
  - 一時停止
  - リセット
- 進捗カード
  - 今日の完了回数
  - 今日の集中時間

### 7.2 円形プログレス

円形タイマーは SVG で実装する。

理由:

- 進捗率の計算がしやすい
- モックに近いリング表現を再現しやすい
- テスト時に描画値の検証がしやすい

## 8. 永続化戦略

### 8.1 初期版

- 設定値は localStorage に保存する
- 今日の進捗も localStorage に保存する

### 8.2 将来拡張

- Flask API を追加する
- SQLite に履歴や設定を保存する
- Repository の実装だけ差し替えて拡張できるようにする

## 9. ユニットテストしやすくするための改善点

今回の設計では、特に以下を守ることでテスト容易性が大きく向上する。

### 9.1 時刻取得を直接書かない

避けるべき例:

- JavaScript 内で Date.now をあちこちで直接呼ぶ
- Python 内で現在時刻 API を直接埋め込む

改善案:

- TimeProvider のような抽象を用意する
- テストでは固定時刻を注入できるようにする

### 9.2 DOM 操作と状態変更を分離する

避けるべき例:

- ボタンクリック関数の中で状態変更と画面書き換えを同時に行う

改善案:

- 状態更新関数は新しい state を返す
- 描画関数は state を受けて DOM を更新する

### 9.3 localStorage へ直接アクセスしない

避けるべき例:

- UI イベントハンドラから直接保存処理を書く

改善案:

- SettingsRepository と ProgressRepository を経由する
- テストではメモリ実装に差し替える

### 9.4 モード遷移ルールを散在させない

避けるべき例:

- 画面更新関数やイベントハンドラごとに if 文で遷移を書く

改善案:

- SessionPolicy に遷移ルールを集約する

### 9.5 setInterval をロジックの中心にしない

避けるべき例:

- 1 秒ごとに残り時間を減らすだけの実装

改善案:

- 終了予定時刻との差分で残り時間を求める
- setInterval は再描画トリガーとして扱う

## 10. テスト戦略

### 10.1 優先して単体テストする対象

- TimerEngine
- SessionPolicy
- ProgressService
- 残り時間計算関数
- 日次リセット判定関数

### 10.2 テストの粒度

- ドメイン層: 純粋関数のユニットテストを中心にする
- アプリケーション層: Repository をモック化してテストする
- Flask 層: ルートが 200 を返すこと、API が期待した JSON を返すことを確認する
- UI 層: DOM テストは必要最小限に留める

### 10.3 代表的なテストケース

- 集中 25 分開始時に終了予定時刻が正しく計算される
- 一時停止後に再開しても残り時間が正しく維持される
- 4 回目の集中完了後に long_break へ遷移する
- リセットでモードと残り時間が初期状態に戻る
- 日付が変わると今日の進捗がリセットされる
- 保存済み設定が再読込される

## 11. 推奨ディレクトリ構成案

  - test_session_policy.py
  - test_progress_service.py

JavaScript をテストする場合は、将来的にフロントエンド用テスト環境を別途導入する。

## 12. 実装順序の提案

1. Flask で単一ページを返す最小構成を作る
2. HTML と CSS で UI モックに近い見た目を作る
3. TimerEngine と SessionPolicy を先に実装する
4. そのロジックに対するユニットテストを書く
5. UI からロジックを呼び出す配線を行う
6. localStorage を利用した Repository を追加する
7. 通知や追加設定などの機能を拡張する

## 13. 結論

このアプリでは、Flask を薄く保ち、フロントエンドに状態管理を持たせる構成が最も実装しやすく、UI モックとの相性も良いです。

同時に、ユニットテストのしやすさを確保するためには、以下の 2 点が重要です。

- タイマー状態遷移ロジックを UI から分離すること
- 時刻取得と保存処理を注入可能な形にすること

この方針により、初期版はシンプルに実装しつつ、将来的な API 化や永続化の強化にも無理なく対応できます。
